前言

Git作为一个开源的分布式版本控制系统，无需服务器端的软件支持，为很多项目的版本维护，更新提供了一个良好的维护系统。

本文章根据一些个人的理解对Git在命令行下的实现以及一些基本的指令操作做了梳理，以此方便自己的学习使用。

*为方便起见，部分内容以github平台的操作为准。

资料

网上关于Git的资料有许多，这里列出几个供平时查阅，同时文中部分内容借鉴于此：

  1. Github官方文档(包括git的使用)

相关概念

HEAD

HEAD 是一个对当前检出记录的符号引用——也就是指向你正在其基础上进行工作的提交记录。

HEAD 总是指向当前分支上最近一次提交记录。大多数修改提交树的 Git 命令都是从改变 HEAD 的指向开始的。

HEAD 通常情况下是指向分支名的。在你提交时，改变了对应分支的状态，这一变化通过 HEAD 变得可见。

从个人理解来看，若整个项目在Git中被组织成一棵树（并非真正的树，很多情况下性质类似），那么每一次提交或者对分支的更改产生、修改、删除新的或者已有的结点，head便是指向某一个结点即提交记录的指向性信息。

结点

由上文所说，项目在Git中的组织类似于树，每一个结点对应于一次提交的记录。

新生成的记录总是基于一个父节点（上一次的提交记录）或者当前指向的分支进行。若指向的结点已经拥有孩子，再次提交相当于生成了一个新的孩子，根据树的性质相当于一个新的分支。

因此对Git来说，提交、修改、删除等操作很大程度上不会真正改动文件，而是改变分支等指向性信息，类似于将树中所有结点打散重构关系。这样使得资源的读取、修改变得更加高效，减少程序对系统的消耗。

分支

分支指向的提交记录与用户操作有关。

远程仓库

远程仓库实际上可以理解为服务器上的分布式仓库，只不过这个仓库是公布在网络上的，本地和其他人都可以访问，借助Git的优势使得你的代码、项目可以分享给其他人甚至让他人帮助你一起进行修改，同时远程仓库也提供了一个项目的备份功能，便于数据的备份和保存。

类似Github等网站便是借助Git工具和网络提供了远程仓库的可视化操作。

相关指令

本地Git操作

git信息管理：

git config <修饰词> <参数> <属性值> | 修饰词可以是--global,表示所有本地仓库均使用此参数

此信息的修改只是表示本地仓库的信息，例如本地仓库的用户和邮箱。修饰词不添加时默认为当前仓库的信息

git config -l | 显示当前所有信息(当前仓库信息和全局信息)

git分支操作：

git add | 将需要文件加入暂存，下次提交包括这些文件

git commit | 提交当前的更改。

git commit –amend | 对当前的结点做一点小的修改，即相当于在其父节点上进行了一些小的改动，之前的提交作废。

git branch | 建立分支。

git branch < 分支名 > < 提交记录名（可使用相对引用） > | 在对应的记录创建分支。

git branch -f A B | 强制修改分支位置，指向一个提交记录，A分支指向B记录。

git checkout | 切换分支。

git checkout < 哈希值 > | 切换至对应哈希值的分支。

git checkout 结点^ | 切换至给定分支的上一个父节点（^本身可以叠加，叠加的数量表示向上的次数）。

git checkout 结点~(数字) | 切换至给定结点的向上（数字）个分支（包括自身）。

^(num) 若当前结点有3个父节点（merge），num可以表示跳转至第几个父节点。

^ ~ 可以单独使用，通过相对引用表示结点而无需通过复杂的提交结点的hash code。

git merge A | 合并（将A快速合并至当前分支），当前分支的结点直接与给定分支A进行合并，两者同时作为新合并的结点的父节点，同时当前分支指向新合成的结点）。

git rebase A | 合并（当前分支合并至A）（LCA，将与给定合并分支A与当前结点的最近公共祖先结点下当前分支上的所有线性的提交记录合并至A，在A上进行不在当前分支上进行）。

git rebase A B | 将B以rebase的方式合并至A，同时当前结点（HEAD）指向B。

撤销变更：

和 提交 一样，撤销变更由底层部分（暂存区的独立文件或者片段）和上层部分（变更到底是通过哪种方式被撤销的）组成。

git reset | 通过分支记录回退几个提交记录实现撤销改动（即改变分支的指向，看上去分支仍指向前几次的记录）。

git revert | reset无法将更改和他人进行共享（本地修改分支的指向），使用revert可以将更改推送到远程仓库与别人进行分享。revert并非直接将结点指向之前的结点，而是新增一个结点表示与该结点对应的结点进行撤销。

git revert < 当前结点 > | 表示撤销对当前提交结点的更改。

整理提交记录（不完全合并）：

git cherry-pick < 提交号 > | 将一些提交复制到当前所在的位置HEAD下面（即不用完全合并，将一些提交放至自己的分支下）。

git rebase –interactive OR -i | 交互式rebase，即当不知道需要进行整理的提交记录的位置或者hash code时，使用交互式rebase，Git会打开UI界面并列出将要被复制到目标分支的备选提交记录，并显示每个提交记录的哈希值和提交说明，一般的UI界面会在文本编辑器中打开一个文件。HEAD3 表示对HEAD3 点（此节点除外）以下对应分支路径上的结点进行操作。

Git Tag：

Git Tag可以永久地将某个特定的提交命名为里程碑，并像分支一样进行引用。塔门并不会随着新的提交而移动，也不能检出到某个标签上面进行修改提交，它就像是提交树上的一个锚点，标识了某个特定的位置。类似于一个重大版本的标记。

git tag < tag名 > | 提交记录标识。

git describe | 用于描述离当前指向的提交记录最近的锚点，即标签。

当需要复制的提交记录为当前分支的祖先结点时，会导致原有的路径被丢弃，使用新的路径。

远程Git操作

远程仓库的连接：

*github自2021年8月13日起取消了直接账户密码登录的登录方式，只能使用令牌(Token)或者SSH/RSA密钥方式进行连接，下面讲述SSH/RSA方式的连接方式

ssh-keygen -t rsa -C <信息> | 生成rsa密钥，适用于ssh方式进行远程仓库的操作

git remote -v | 查看当前仓库使用的提交地址。https协议表示使用令牌进行连接，git@github.com:user/gitreponame.git 表示使用SSH连接的方式进行远程仓库的更改

git remote set-url origin ssh仓库地址 | 设置远程连接使用SSH/RSA方式，需要注意的是，若未特别指定，此后进行连接会默认以~/.ssh/id_rsa作为私钥进行连接。

将公钥(keyname.pub)存放至远程仓库，可实现SSH密钥连接进行远程仓库的提交

创建远程仓库：

git clone < 对应仓库的地址 > | 在本地创建一个对应远程仓库的拷贝。

clone来的本地仓库具有远程分支，反映了远程仓库在上一次和其通信的状态。

远程分支的命名格式是：< 远程仓库名 >/< 分支名 >。

远程仓库的通信：

git fetch < 仓库地址，http协议或git > | 和远程仓库通信并更新当前指向的分支状态，可以理解为单纯的下载文件，但是本地的分支状态并没有进行改变，比如本地的main分支指向的提交结点无改变。

可以理解为在本地仓库中存在本地分支和远程分支两种分支，由于是分布式版本管理，相当于本地和远程同时共存但是又相互分开，本地归本地操作管，远程与对应的远程仓库相关联，本地的修改并不影响（相当于自己的单方面改动是无法修改远程仓库，只有进行和远程仓库的通信并与本地的进行通信才可以进行改动）。

参数：

git fetch < origin > < branch > | git会到远程仓库的对应分支上，将所有本地不存在的分支放至对应分支上。

远程仓库到本地的合并：

git pull < 远程仓库的合并 > | 相当于先进行拉取再与本地对应的分支进行合并，即git fetch和git merge的结合（所以git fetch相当于只是下载了文件）。

由于是和本地当前结点进行合并，所以要注意两个分支之间的关联。

本地到远程仓库的合并：

将自己的本地分支合并分享至远程仓库，发布自己的成果。

git push < 远程仓库地址 > | 将更改提交至远程仓库上。

git push 将自己的变更上传到指定的远程仓库，在远程仓库上合并你的提交。

git push 不带参数时其行为与Git的push.default的配置有关，在使用该命令的时候需要修改一下配置。

git push使用时：

  1. 首先将对应的本地分支合并至远程对应的分支上。
  2. 远程仓库的分支状态改变。
  3. 本地的远程分支状态随远程仓库的变化一起进行变化。

参数：

git push < remote > < place > | 切换至本地place分支，获取所有的提交，在到远程仓库remote中找到place分支，将本地对应的提交记录添加至远程仓库对应分支（当前本地HEAD指向不进行变动，即不变动本地的提交记录的指向）。

远程分支版本回退：

*对于个人的远程分支可以这样粗暴点（

git reflog | 查看当前提交结点前全部的提交id(自己的提交)

git reset id | 回退

git push -f | 远程强制推送

多人协同的本地至仓库提交：

当不另设新的分支且无需从某个特定结点进行提交操作时：

  1. 使用git pull –rebase或git pull或git merge使本地的提交基于远程最新的提交结点。
  2. 本地进行操作。
  3. 使用git push进行提交。

远程与本地的分支关联（跟踪）：

在进行git push或者git pull时候，只有两个分支有关联（在树上表示为处在同一个路径上），此时两个分支才能进行对应的操作。默认情况下main对应于远程的o/main分支，因此在进行操作时默认两个分支进行操作，我们可以设置o/main远程分支关联与本地的自定义分支，使得两个分支能进行相应的操作。

git checkout -b mine o/main | 使本地mine分支与远程的main分支进行关联（跟踪）。

git branch -u o/main mine | 与上一个指令相同，若省略第二个参数则默认当前本地分支与对应的远程分支进行关联。
