仅仅记录个人学习java时学到的知识和心得

类

由类定义一个具体的对象的组织和其上的操作。 当声明一个具体的存在对象时，类在其上的定义可以帮助辨识这个存在的组织结构和操作。

对象分为实际的类别和引用的类别。

JAVA中的变量实际上都是引用，类似于C中的指针为指向性信息。
不同的是，这些指向性信息的类别不一定和指向的对象的真实类别相符合，这也就实现JAVA的一个特性：多态。

字段(field)

对应于方法的属性，表明了一个类相关方面的属性。

在class中定义的字段称为实例字段，其特点是每个实例都有独立的字段且占用不同的空间，互不影响。
但是由static修饰的字段为静态字段，只占有一个”共享“空间，且所有实例共享该字段。
更改一个实例的静态字段会改变与之相同类的所有实例的字段值，该静态字段实际上指向的都是类的静态字段，因此在访问的时候尽量使用类名.静态字段

接口

抽象方法本质上是定义了接口(interface)规范，接口可以说，比如一个对象是一个具体类的实例(instance)，那么其方法可以是这个对象或者说这个类的接口，提供了类(对象)与外界(系统，用户)的通信，接口类即只提供了抽象方法的类，只说明了有哪些通信的方法，这为子类的继承提供了规范。
可以理解为是相关事物的全集，比如对所有人类来说的全集是人类这个物种(从一个角度进行说明)，则人类这个类就是一个接口，声明了人类有那些行为，具体的需要人单独的个体(对象，instance)进行说明。

接口类(interface)不能有字段，当用interface声明一个接口(interface name)时，不能有字段，同时所有的方法默认为public abstract。

此时子类不再用extends而是implements，extends只能继承一个类，而implements可以是实现同时继承多个interface(用’,’隔开)。

JAVA的接口(interface)特指其定义，表示一个接口类型和一组方法签名。

接口可以继承(extends)自另一个接口。

抽象程度

接口(interface)>抽象类(abstract)>实例(具体指明的类)(instance)

接口的静态方法

接口可以定义静态字段，此时静态字段得要final类型进行修饰，因为只能是这样，因此实际上interface可以定义字段，不过会自动使用public static final修饰

default方法

接口可以定义default方法(类似于lambda)，即接口定义一个拥有具体执行步骤的方法(类似于所有的子类和示例都具有完全一致的方法，那么可以直接在接口中进行定义，比如全体人类都使用肺进行呼吸)。
default方法的目的是，当我们需要给接口新增一个方法时，会涉及到修改全部子类。如果新增的是default方法，那么子类就不必全部修改，只需要在需要覆写的地方去覆写新增方法。

静态方法

与静态字段相似的方法，使用类名就可以进行调用。

静态方法通常用于工具类和辅助方法。

this

this表示类当前被调用的实例

*静态方法无法访问实例字段，只能访问静态字段（static）

包(package)

用于管理类，当它人要使用类和接口时，使用包能更好地进行区分并且不会造成类名相同导致的混乱。

在每一个类文件前加上package [包名]，表示当前文件中的类只能被包名.类名调用。

包无父子关系，但所有java文件的目录层次要和包层次相同。以’.’作为分割。

位于同一个包的类，可以访问包作用域的字段和方法。不用public、protected、private修饰的字段和方法就是包作用域。

bin目录主要存放的是已经编译好的java文件以便调用，可以使用javac -d bin目录 文件1 文件2……文件n 自动进行编译。

使用import进行引用，在编写class时，编译器(JAVAC)会自动做两个动作：

  1. 默认自动import当前package的其他class
  2. 默认自动import java.lang.*

作用域

即可以被访问的范围。

public的class和interface可以被其他任何类访问（需要import）。

public的field和method可以被访问但是需要有访问对应class的权限。

private的field和method无法被其他类访问，在进行编写的时候尽量把public方法写在前面。

Java支持嵌套类，那么嵌套类可以访问private。

包作用域(package)指一个类可以访问同一个包中没有被public，private修饰的class以及没有public，protected，private修饰的字段和方法。

局部变量指的是在方法内部定义的变量，其作用域从变量声明处到对应的块（{}）结束处。

final修饰符(相当于一个常量)可以：

  1. 阻止class被继承。
  2. 阻止子类覆写class的方法。
  3. 阻止field被重新赋值。
  4. 阻止局部变量被重新赋值。

一个.java文件只能包含一个public类，但可以包含多个非public类。

内部类

定义在class内部的class称为嵌套类(nested class)。

在实例化的时候需要类.new 嵌套类()

匿名类

因为通常内部类不关心类名，只需去调用，因此使用匿名类。匿名类也可以继承自普通类。

使用static修饰的内部类，不依附于其他类而是一个独立的类，仍然可以访问外部类的private字段和静态方法。

即内部类分为三种：

  1. 内部普通类
  2. 匿名类
  3. 静态内部类

JVM用于提供环境运行class文件

classpath的设定：

  1. 在系统环境变量中进行设定。**(不推荐)**

若没有设定，默认为当前目录。若在IDE中，则自动传入当前工程的bin目录和引入的jar包。

jar包

比如有多个class文件，散落于各级目录，若每次调用都使用参数一一填入会相当麻烦，因此引入jar包，可以把package组织的目录层级以及各个目录下的文件打包成一个jar文件。

jar包实际是一个zip格式的压缩文件，相当于目录，因此我们可以把项目打包成jar包，这样我们可以使用JVM运行的时候指定classpath为jar包，方便调用其中的类和其他事物(jar包记录了package的组织层次，相当于一次性制定了classpath的目录)

jar的制作，找到正确的目录将其打包成zip文件并更改后缀名便可。jar包的是第一层目录应该为bin下(存放class文件的目录)各个package的第一层目录。

由于jar文件本身是多个class文件的集合，因此在运行的时候需要指定一个入口class(相当于整个程序的入口)，这是我们可以的使用/META-INF/MANIFEST.MF文件指定Main-Class和其他信息，放至jar包中，JVM会自动进行读取。

此时可以使用java -jar name.jar运行。

实际上为java -cp name.jar Main-Class。

rt.jar

JVM自带的Java标准库也是以Jar文件的形式进行存放，即rt.jar

因此在java9之前，自己编写的jar文件中会依赖很多其他的库(jar)文件，导致运行时要一一进行指明。

模块

为了解决依赖这个问题，从Java9开始引入了模块，让class容器能自带”依赖关系“，使得编译运行时能自动定位并解决依赖关系。

因此rt.jar被拆分成几十个模块并以.jmod拓展名标识。模块名即是文件名。

在jvm标准库中的模块文件中，模块之间的依赖关系已经由module-info.class文件写入，所有的模块之间或者间接依赖java.base模块。

使用方法

首先建立工程文件，在src源码下构建module-info.java文件(文件名只能是此)，之后在其中编写模块依赖。

进行编译，使用代码在当前目录中使用

javac -d bin /src/module-info.java 已有的源码

打包成jar，使用

jar –create –file name.jar –main-class 指定入口类 -C bin

即在当前目录得到对应的jar包

*jar包可以设置classpath，运行的时候自动加载

若要转换成模块，则我们使用

jmod create –class-path name.jar name.jmod

得到我们自己定义的模块

打包jre，即得到我们自己的java运行环境，因为在实际的应用中，我们并非完全需要jvm中所有的标准库文件，因此我们可以通过使用自己的模块来构建一个jre环境，换句话来说，得到了一个可以精简版的jre环境，且这个环境可以给别人使用并且给我们的程序“瘦身”

静态工厂方法

Integer.valueOf()是Integer包装类的一个方法，在Java中进行一定的优化，比如对较小的数不会重新再构建一个实例而是返回值相同的已有实例，能节省时间和空间（内部缓存优化），因此在创建Integer实例时，我们尽量可以使用Interger.valueOf()进行创建，这样有助于节省内存等。

<p>static代表类层次,类中的静态变量、方法或代码块并不具体到实例，它们可以与类的名称一起使用，而无需创建类的对象来访问类的静态特性。</p> <p>继承是发展进化的基础，extends用于继承，super代表父类对象。</p> <p>final 修饰符可应用于类、方法，代表进化终止。修饰变量代表无法再改变。</p> title="前言"></a>前言</h2><p>Git作为一个开源的分布式版本控制系统，无需服务器端的软件支持，为很多项目的版本维护，更新提供了一个良好的维护系统。</p> <p>本文章根据一些个人的理解对Git在命令行下的实现以及一些基本的指令操作做了梳理，以此方便自己的学习使用。</p> <p>*为方便起见，部分内容以github平台的操作为准。</p> <h3 id="资料"><a id="相关概念"><a href="#相关概念" class="headerlink" title="相关概念"></a>相关概念</h2><h3 id="HEAD"><a href="#HEAD" class="headerlink" title="HEAD"></a>HEAD</h3><p>HEAD 是一个对当前检出记录的符号引用——也就是指向你正在其基础上进行工作的提交记录。</p> <p>HEAD 总是指向当前分支上最近一次提交记录。大多数修改提交树的 Git 命令都是从改变 HEAD 的指向开始的。</p> <p>HEAD 通常情况下是指向分支名的。在你提交时，改变了对应分支的状态，这一变化通过 HEAD 变得可见。</p> <p>从个人理解来看，若整个项目在Git中被组织成一棵树（并非真正的树，很多情况下性质类似），那么每一次提交或者对分支的更改产生、修改、删除新的或者已有的结点，head便是指向某一个结点即提交记录的指向性信息。</p> <h3 id="结点"><a href="#结点" class="headerlink" title="结点"></a>结点</h3><p>由<a href="#head">上文</a>所说，项目在Git中的组织类似于树，每一个结点对应于一次提交的记录。</p> <p>新生成的记录总是基于一个父节点（上一次的提交记录）或者当前指向的分支进行。若指向的结点已经拥有孩子，再次提交相当于生成了一个新的孩子，根据树的性质相当于一个新的分支。</p> <p>因此对Git来说，提交、修改、删除等操作很大程度上不会真正改动文件，而是改变分支等指向性信息，类似于将树中所有结点打散重构关系。这样使得资源的读取、修改变得更加高效，减少程序对系统的消耗。</p> <h3 id="分支"><a href="#分支" class="headerlink" title="分支"></a>分支</h3><p>分支指向的提交记录与用户操作有关。</p> <h3 fetch相当于只是下载了文件）。</p> </blockquote> <p>由于是和本地当前结点进行合并，所以要注意两个分支之间的关联。</p> <p><strong>本地到远程仓库的合并：</strong></p> <p>将自己的本地分支合并分享至远程仓库，发布自己的成果。</p> <blockquote> <p>git push < 远程仓库地址 > | 将更改提交至远程仓库上。</p> </blockquote> <p><code>git push</code> 将自己的变更上传到指定的远程仓库，在远程仓库上合并你的提交。</p> <p><code>git push</code> 不带参数时其行为与Git的push.default的配置有关，在使用该命令的时候需要修改一下配置。</p> <p>git push使用时：</p> <ol> <li>首先将对应的本地分支合并至远程对应的分支上。</li> <li>远程仓库的分支状态改变。</li> <li>本地的远程分支状态随远程仓库的变化一起进行变化。</li> </ol> <p>参数：</p> <blockquote> <p>git push < remote > < place > | 切换至本地place分支，获取所有的提交，在到远程仓库remote中找到place分支，将本地对应的提交记录添加至远程仓库对应分支（当前本地HEAD指向不进行变动，即不变动本地的提交记录的指向）。</p> </blockquote> <p><strong>远程分支版本回退：</strong></p> <p>*对于个人的远程分支可以这样粗暴点（</p> <blockquote> <p>git reflog | 查看当前提交结点前全部的提交id(自己的提交)</p> </blockquote> <blockquote> <p>git reset id | 回退</p> </blockquote> <blockquote> <p>git push -f | 远程强制推送</p> </blockquote> <p><strong>多人协同的本地至仓库提交：</strong></p> <p>当不另设新的分支且无需从某个特定结点进行提交操作时：</p> <ol> <li>使用git pull –rebase或git pull或git merge使本地的提交基于远程最新的提交结点。</li> <li>本地进行操作。</li> <li>使用git push进行提交。</li> </ol> <p><strong>远程与本地的分支关联（跟踪）：</strong></p> <p>在进行git push或者git pull时候，只有两个分支有关联（在树上表示为处在同一个路径上），此时两个分支才能进行对应的操作。默认情况下main对应于远程的o/main分支，因此在进行操作时默认两个分支进行操作，我们可以设置o/main远程分支关联与本地的自定义分支，使得两个分支能进行相应的操作。</p> <blockquote> <p>git checkout -b mine o/main | 使本地mine分支与远程的main分支进行关联（跟踪）。</p> </blockquote> <blockquote> <p>git branch -u o/main mine | 与上一个指令相同，若省略第二个参数则默认当前本地分支与对应的远程分支进行关联。</p> </blockquote> </div> <footer class="article-footer"> <a 流浪者日志002<br>date: 2021-09-07 10:32:30<br>tags: 银河系流浪者日志 </p> <hr> <p>“指令在能量的驱动下通过不同的排列组合组织不同的物质、事物状态并随着时间和空间发生转换”</p> <p>指令，对于计算机而言，可以说是用户与计算机之间交流沟通的桥梁。<br>指令以一种人类或计算机用户能够理解的形式，将用户对所需描述现实对象以计算机能够理解的形式传递给计算机。<br>计算机在对“指令”进行理解并执行相应动作。</p> <p>今日在上语言课时受老师启发，指令是否也可以更加广泛。</p> <p>“特修斯之船”这一哲学问题中指出如果当初一个被命名为“特修斯之船”的木船，在经过常年累月的航行和维修后，是否还是原来的船。</p> <p>课上讨论的结果是“特修斯之船”本就是用于描述这一可以载人，可以航行的木船这一事物而并非构成木船的零部件，是其抽象的功能，而并非零部件。</p> <p>木船历经的风霜也就是这一事物在程序（由航行，停靠港口，维修组成的指令序列）的指导下随着时间空间而发生的状态变化过程，<br>而这一过程需要大海、人类的能量助力。</p> <p>可以说，处在时间轴上看，木船就是一个进程，一个基于时间表征的发生在时空间的状态转移过程。</p> <p>换句话来说，船作为一类事物，特修斯之船就是它的一个实例。字段的更改并不会影响这个实例“是否还是原来的实例”。</p> <p>我喜欢语言，因此我第一个想到是用语言的意义来解释这个问题。<br>假如词语是用于链接现实事物的，那么它就得被群众或是相关的人所信任，且现实事物也是被大众而非个人所认可的单一事物。<br>例如“996”对于苦受压榨的程序员来说是一种病态的工作模式，但对于世界上某位正在解答数学问题的人而言，“996”或许就是一个简单的数字。<br>“特修斯之船”所指代的船为什么叫“特修斯之船”，是因为建造它的人抑或是买家因为某种原因给他命了名，再将这个名字传递给大众，刻在船身上，登记在船只登记手册里。<br>可以说“特修斯之船”的指代本身就是被大众所信赖认可的，人们更关注这个名字和这艘船本身的联系而并非和零部件的联系，人们也知道这是他们面前这一艘船的名字，<br>而并非哲学家嘴中的零部件。</p> <p>前一种着眼于木船本身具有的性质和他作为一个事物整体的功能，后一种解释注重于语言的意义以及大众的观点。感觉我的观点有点玩文字游戏:)</p> <p>不过就以前一种而言，指令是否就存在于我们的生活中，抑或是宇宙本身也是由指令驱使？</p> <p>在《复杂的引擎》一书里，指令是驱使物质组织的规则，或者说，指令与规则本身就是一类东西。</p> <p>试想一下，构成人类的DNA，计算机芯片的指令集，都可以算是各种情景下的规则，而我们将这些规则排列组合：<br>DNA的组合形成了70多亿个独一无二的人类个体，计算机芯片的指令的组合造就了现在的计算机世界。<br>规则构成的程序在能量的驱动下（人类需要生物能，计算机需要电能），组织对应事物在发展着的时空间中发生着变化（状态上的改变）</p> <p>这些规则不仅局限于计算机指令，也可以是人类社会的律法，更可以是物理规则乃至数学规则。</p> <p>而这些规则运用的产物-“进程”-也构成了更大的系统（环境）：人类社会，地球，银河系，整片宇宙。</p> <p>这是“指令”的奥秘。</p> <p>后话：也许这不是这个世界的实质（如果有的话），也许只是一个相对概念，但是我们追求的不应当是“终极真理”，而应当是适用于当下情况、领域的理论。<br>那么，现如今我们一直追求的“大统一”理论是否只是人类强加在这个世界上的臆想？我现在没有这个能力解答。不过今天我倒是有个关于引力的想法：</p> <p>计算机之间的进程可以互相传递消息，将人类视为一个进程，那么人类社会是进程间相互作用的产物，所有的星系相互吸引，构成了超星系。</p> <p>贯穿这其中的始终有能量，尽管形式并不完全一致，那么引力是否就是能量的一种表现形式？抑或是能量本身只是人类对驱动事物这种现象的描述，只是正好能够契合物理现象而已？</p> <p>那么存在又是什么？</p> </div> <footer
